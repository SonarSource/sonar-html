Why is this an issue?

Non-interactive DOM elements should usually not be given interactive ARIA roles. Doing so changes how assistive technologies announce the element without giving it the native behavior users expect.

For example, <div role="button"> is announced as a button, but it does not automatically get the keyboard support, focus behavior, or activation semantics of a real <button>. This mismatch can confuse users and make the page less accessible.

In most cases, prefer native interactive elements instead of assigning interactive roles to non-interactive markup.

Exceptions

The ARIA in HTML specification allows some non-interactive elements to take interactive roles when they are used to implement composite widgets.

Examples include:

  • <fieldset role="radiogroup">
  • <ul> or <ol> with roles such as listbox, menu, menubar, radiogroup, tablist, or tree
  • <li> with roles such as menuitem, menuitemcheckbox, menuitemradio, option, row, tab, or treeitem
  • <table role="grid">

These exceptions are valid only when the markup implements the corresponding ARIA composite-widget pattern.

How to fix it

Use a native interactive element whenever possible. For example, replace <li role="button"> with a real <button> when the content is meant to behave like a button.

Keep an interactive ARIA role on a non-interactive element only when the ARIA in HTML specification explicitly allows that composite-widget pairing, such as <fieldset role="radiogroup"> or <ul role="tablist">.

Code examples

Noncompliant code example

<li role="button">Foo</li> <!--Noncompliant; "li" isn't interactive, but "button" is-->

Compliant solution

<button type="button">Foo</button>
