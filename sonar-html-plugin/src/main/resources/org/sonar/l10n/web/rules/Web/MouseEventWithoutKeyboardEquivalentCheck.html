Why is this an issue?

Offering the same experience with the mouse and the keyboard allows users to pick their preferred device.

Additionally, users of assistive technology will be able to browse the site even if they cannot use the mouse.

This rule raises an issue when:

  • an HTML element with an onClick attribute doesn’t also have an onKeyDown or onKeyUp attribute.
  • an HTML element with an onMouseover attribute doesn’t also have an onFocus attribute.
  • an HTML element with an onMouseout attribute doesn’t also have an onBlur attribute.

In the case of onClick, the equivalent keyboard handler should activate on the "Enter" and "Space" keys, since these are the keys assistive technologies use to activate buttons.

onKeyPress is also recognised as a keyboard equivalent for backward compatibility, but the event is deprecated and should not be used in new code; prefer onKeyDown.

Noncompliant code example

<div onClick="doSomething();" ...>                   <!-- Noncompliant - 'onKeyDown' or 'onKeyUp' missing -->
<a onMouseover="highlight();" ...>                   <!-- Noncompliant - 'onFocus' missing -->
<a onMouseout="unhighlight();" ...>                  <!-- Noncompliant - 'onBlur' missing -->

Compliant solution

The most accessible fix is to use the native interactive element. A <button> is focusable, exposes the right role to assistive technology, and is activated by both Enter and Space without extra code:

<button onClick="doSomething();">Do something</button>      <!-- Compliant -->

If a custom element must remain non-interactive (e.g. a <div>), the equivalent keyboard handler must check the key so it only triggers on Enter or Space, and the element must expose role="button" and tabindex="0" so it is announced and reachable:

<div role="button" tabindex="0" onClick="doSomething();"
     onKeyDown="if (event.key === 'Enter' || event.key === ' ') doSomething();">   <!-- Compliant -->
</div>

For purely visual hover effects, prefer CSS over scripted mouse handlers — the :hover and :focus-visible pseudo-classes give mouse and keyboard parity without any JavaScript:

<a href="/page" class="link-with-hover">Link</a>   <!-- Compliant - hover styling handled by CSS :hover / :focus-visible -->

When scripted hover behaviour is genuinely needed, pair onMouseover / onMouseout with onFocus / onBlur:

<a href="/page" onMouseover="highlight();" onFocus="highlight();"
                onMouseout="unhighlight();" onBlur="unhighlight();">Link</a>  <!-- Compliant -->

Exceptions

The following native interactive elements are skipped when they carry an onClick attribute, because keyboard activation is built into the platform:

  • <input type="button"> and <input type="submit">,
  • <button>,
  • <a>,
  • <summary>.

Be aware that the anchor exception is broad: only an <a> with an href attribute is actually focusable and activated by Enter — an <a onClick> without href behaves like inert text from the keyboard. For button-like actions inside the page, prefer <button> over a bare <a>.

An issue will still be raised for any element that carries role="button" (including <button role="button">), because the explicit role overrides the implicit behaviour and the element is treated as a custom widget.

Custom button-like web components

Design systems often ship custom elements that behave like a <button> (e.g. lightning-button, ion-button, clr-button). Listing them in the whitelistedElements parameter suppresses the issue for those tag names:

whitelistedElements = lightning-button,lightning-button-icon,clr-button,ion-button

The default value covers the Salesforce Lightning lightning-button* family.
