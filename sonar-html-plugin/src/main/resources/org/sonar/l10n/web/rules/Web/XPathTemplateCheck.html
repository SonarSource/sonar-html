Why is this an issue?

This rule allows the definition of custom rules using XPath expressions. It follows the same principles as the XML XPath rule (RSPEC-140), but adapted for HTML documents.

This HTML rule is fully compatible with the XML XPath rule concepts and behavior:

  • Uses the same javax.xml.xpath API for evaluation
  • Creates line issues for node matches
  • Creates file issues for boolean expressions
  • Supports file pattern filtering
  • Follows the same remediation approach

Issues are created depending on the return value of the XPath expression. If the XPath expression returns:

  • a single or list of HTML nodes, then a line issue with the given message is created for each node
  • a boolean, then a file issue with the given message is created only if the boolean is true
  • anything else, no issue is created

Here is an example of an XPath expression to log an issue on each 'img' tag missing the 'alt' attribute:

//img[not(@alt)]

Limitations

  • XPath 1.0 only — Functions like matches() from XPath 2.0 are not available.
  • Namespace-unaware — The HTML DOM is parsed without namespace awareness. Namespace-prefixed XPath expressions (e.g., //xhtml:div) are not supported.
  • Use descendant axis — The internal DOM wraps elements in a synthetic root node. Use descendant expressions like //img instead of absolute paths like /html/body/img.
  • Element matches only — Only XPath expressions matching elements create line-level issues. Expressions matching attributes or text nodes (e.g., //img/@alt) will not produce issues.
